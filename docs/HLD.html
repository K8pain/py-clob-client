Reader journey

Primero entiendes el problema, luego los flows, después los constraints técnicos, y recién ahí el plan de trabajo. Esa secuencia hace la doc más humana.

1. Qué estamos construyendo

Un SDK Python para operar el CLOB de Polymarket de forma segura, tipada y razonablemente ergonómica. El objetivo es esconder complejidad operativa sin esconder conceptos importantes.

Zoom out

Qué es

Una fachada de alto nivel para consultar mercado, autenticarse por niveles, firmar órdenes y ejecutar workflows RFQ.

Para quién

Developers de bots, integradores backend y equipos que automatizan trading o market data sobre Polymarket.

Qué problema resuelve

Reduce fricción en auth, firmas, rounding, payloads y conexión entre wallet, credenciales API y endpoints.

Principio rector: distill the model. Quitar duplicación y hacer explícitos solo los conceptos que realmente cambian el comportamiento.

2. User experience / Developer experience

No hay UI tradicional; la experiencia del usuario es la ergonomía de la API Python, la legibilidad de los ejemplos y la previsibilidad de los errores.

Zoom in

Happy flows

  • Lectura: crear `ClobClient` en L0 y consultar salud, tiempo, orderbooks y mercados.
  • Trading: crear o derivar credenciales, construir orden, firmar y publicar.
  • RFQ: entrar por `client.rfq`, crear request, cotizar y cerrar el loop de aprobación.

Alternative flows

  • Wallet proxy con `signature_type` y `funder`.
  • FOK sin liquidez suficiente.
  • Resolución dinámica y caché de tick size antes de firmar.
  • Errores por credenciales ausentes o inválidas en L2.

3. Necesidades técnicas

El diseño técnico tiene que preservar exactitud criptográfica y simplicidad de uso. En este repo, los errores pequeños en auth o rounding tienen impacto alto.

Conceptos principales

ClobClientFachada principal, compone signer, builder, credenciales y RFQ client.
SignerEncapsula private key y chain para firma.
OrderBuilderTraduce intent de orden a `SignedOrder` usando rounding por tick size.
RfqClientExtiende el cliente con workflows request/quote/approve.

Reglas técnicas críticas

  • Auth por niveles: L0 público, L1 con firma wallet, L2 con API creds + request signing.
  • Rounding: depende de `ROUNDING_CONFIG` y del `tick_size` resuelto para el token.
  • Serialización: JSON compacto y estable para evitar firmas inconsistentes.
  • Sin DB local: es un SDK puro; no requiere migraciones internas.
Dependencias externas`py_order_utils` y `py_builder_signing_sdk` concentran parte de la lógica de firma y builder config.
Patrones de mantenibilidadPreferir funciones pequeñas, dataclasses, separación entre crear y usar dependencias, y edge cases documentados.

4. Testing & security

La testabilidad no es un nice-to-have: es el mecanismo que impide romper firma, auth y precisión numérica al iterar rápido.

Cobertura objetivo

  • Headers y signing helpers.
  • Order builder y conversiones.
  • RFQ params y payload formatting.

Checks de seguridad

  • No loggear secrets.
  • Validar `side`, `chain_id`, `tick_size`.
  • Mantener serialización determinística.

Side effects

  • Cambios en headers rompen endpoints privados.
  • Cambios en rounding alteran fills y precios efectivos.
  • Cambios en config afectan addresses on-chain.
Cuando se toque auth o firma, asumir impacto de seguridad alto y recomendar una revisión explícita.

5. Plan de trabajo

La estrategia recomendada es MVP-first: primero interfaz y contrato, luego implementación mínima, luego tests, ejemplos y refinamiento.

1

2
Implementar en el módulo correcto
0.5 día. Preferir `ClobClient` o `RfqClient` antes de expandir superficie innecesaria.
3
Añadir tests
0.5–1 día. Cobertura de paths felices y edge cases.
4
Documentar y ejemplificar
0.5 día. README/examples/docs alineados.
Definition of Done
Método funcional + tests relevantes + ejemplo + docs.
Riesgo principal
Cambios upstream en la API CLOB o en contratos implícitos de payload.

6. Ripple effects

En SDKs, el cambio de una función pública rara vez termina en una sola función. Hay ejemplos, README, integradores externos y expectativas de compatibilidad.

  • Actualizar README cuando cambie la API pública.
  • Sincronizar ejemplos ejecutables con los nuevos contratos.
  • Comunicar breaking changes en firma, tipos o auth.
  • Verificar compatibilidad con bots o backends consumidores.

7. Broader context

El repo es útil porque está enfocado. Esa especialización también marca sus límites y su espacio de evolución.

Limitaciones actuales

  • Acoplamiento fuerte a Polymarket CLOB.
  • Tipado parcial de respuestas remotas.
  • Errores de red dependen de helpers HTTP sin capa avanzada de resiliencia.
