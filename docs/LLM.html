LLM Working Document

Una guía operativa para asistentes que necesitan ser útiles, no ruidosos.

Este documento le da a un LLM un mapa claro del repo: qué importa, qué no conviene tocar sin cuidado, qué patrones seguir y cómo detectar ripple effects antes de que aparezcan en producción.

Ideal outcome

Que el siguiente asistente o mantenedor pueda leer esto y acertar más rápido, con menos overengineering y menos riesgo de romper el core.

1. Qué debe entender un LLM

Este repo no necesita un asistente que “haga mucho”, sino uno que haga cambios pequeños, consistentes y seguros. La misión es reducir errores, no aumentar superficie.

Operating model

Producto

SDK Python para CLOB de Polymarket con lectura de mercado, auth multinivel, ordenes firmadas y RFQ.

Usuario

Bots, backends e integradores que valoran contratos estables y baja fricción operativa.

Meta del asistente

Modificar lo mínimo posible para resolver lo necesario, con tests y docs alineadas.

2. UX/DX esperada en cada contribución

La experiencia del usuario del SDK depende de que los métodos nuevos parezcan naturales dentro del cliente existente.

Happy path del contribuyente

  1. Leer un método análogo ya existente.
  2. Extender la API pública sin inventar otra puerta de entrada innecesaria.
  3. Añadir test unitario o regresión.
  4. Actualizar ejemplo o documento si cambió comportamiento observable.

Flujos alternativos

  • Si la auth cambia, conservar compatibilidad L0/L1/L2.
  • Si la serialización cambia, preservar determinismo para firmas.
  • Si el endpoint es incierto, aislar el cambio en helper pequeño y documentar límites.

3. Reglas técnicas de implementación

El LLM debería preferir función sobre clase, reuso sobre expansión y precisión sobre “abstracción creativa”.

Reglas de diseño

  • Funciones pequeñas y enfocadas.
  • Dataclasses y tipos existentes antes de crear modelos nuevos.
  • Separar creación de dependencias y uso de dependencias.
  • Reusar `http_helpers`, `headers` y módulos de builder para no divergir.
Heurística útil: si una idea nueva no simplifica el repo, probablemente sobra.

Datos duros que no conviene olvidar

Tick sizes`0.1`, `0.01`, `0.001`, `0.0001`.
BUY / SELLCambian maker/taker amounts y assets en RFQ.
L2Depende de `ApiCreds` y firma consistente del request.
Dependencias`py_order_utils` y `py_builder_signing_sdk` son parte de la cadena crítica.

4. Testing y seguridad

Cada cambio debería venir con mentalidad de tester: ¿qué edge case rompe esto?, ¿qué estado inválido conviene bloquear antes?

Checklist mínimo

  • Test unitario del comportamiento nuevo.
  • Regresión si se toca auth o rounding.
  • Docs o ejemplo actualizados si cambia la API pública.

Edge cases obligatorios

  • Inputs inválidos.
  • Sin liquidez para FOK.
  • Fallos de red/timeouts.
  • Credenciales ausentes.

Seguridad

  • No imprimir secrets.
  • Preservar serialización determinística.
  • Validar auth temprano.

5. Workflow recomendado para el asistente

Un buen cambio en este repo suele seguir una ruta muy corta. Cuando una solución se ve demasiado sofisticada, probablemente ya va tarde.

A
Leer contexto mínimo
Revisar archivos análogos y puntos de entrada públicos existentes.
B
Hacer el cambio mínimo
Evitar refactors laterales si no son necesarios para la tarea.
C
Probar lo relevante
Suite focalizada primero; suite amplia solo si el cambio toca core behavior.
D
Actualizar comunicación
Docs, ejemplos y notas para consumidores si hubo ripple effects.

6. Ripple effects que el LLM no debe olvidar

Los cambios útiles rara vez viven aislados. El asistente tiene que pensar un paso más allá del patch.

README
Si cambió la manera de usar la API, el snippet público también debe cambiar.
Examples
Los ejemplos son la UX real del repo; deben seguir siendo copy-pasteables.
Tests
Headers, signing, order_builder y RFQ son zonas sensibles.
Release notes
Si hay cambio breaking o visible, conviene dejarlo explícito.

7. Contexto amplio y evolución futura

Entender limitaciones actuales ayuda a producir cambios más realistas y a detectar dónde sí vale la pena invertir.

Limitaciones actuales

  • Especialización en Polymarket CLOB.
  • Respuestas remotas con tipado parcial.
  • Resiliencia de red mejorable.
Future 1
Cliente con respuestas más estrictamente tipadas.
Future 2
Retries configurables por endpoint.
Future 3
Tracing/logging estructurado opcional para observabilidad.
Closing note El objetivo no es volver al asistente más creativo; es volverlo más preciso, más disciplinado y más fácil de revisar.
